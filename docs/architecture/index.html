Architecture Decisions

Every structural decision in this project is documented here with the reason behind it. Architecture is not decided by instinct — it is decided consciously and recorded.

When we make a new structural decision — where a file goes, why a component is split, why a folder was created — it gets recorded here before or immediately after the commit.

Entry Format

Decision: What we decided.

Session: When we decided it.

Why: The reasoning.

Alternatives considered: What else we could have done.

Consequence: What this decision requires or prevents in the future.

ADR-001 — Plain JavaScript (no TypeScript) Pre-curriculum

Decision: Use plain JavaScript throughout the entire curriculum. No TypeScript.

Session: Pre-curriculum (project initialization)

Why: Adding TypeScript types introduces a second mental model on top of React's mental model. Students learning both simultaneously will attribute confusion to the wrong thing. Type errors will distract from the React concept being taught.

Alternatives considered: TypeScript from the start (common in professional environments); adding TypeScript in Layer 6 as an architecture topic.

Consequence: We do not add TypeScript. If this is revisited, it becomes its own session with its own session doc. No implicit TypeScript introduced via library types.

ADR-002 — npm as package manager Pre-curriculum

Decision: Use npm. Not pnpm, not yarn.

Session: Pre-curriculum (student preference)

Why: Fewest moving parts. npm comes with Node.js — no additional install. Reduces environment variables. Students can Google npm errors more easily.

Alternatives considered: pnpm (faster, disk-efficient, stricter); yarn (historically popular, slightly nicer output).

Consequence: All install commands use npm install. Lock file is package-lock.json. Scripts run with npm run.

ADR-003 — No CSS frameworks or CSS-in-JS Pre-curriculum

Decision: Use plain CSS only. No Tailwind, no CSS Modules, no styled-components.

Session: Pre-curriculum (curriculum design)

Why: CSS frameworks hide what CSS is doing. We need to understand the cascade, specificity, and box model before reaching for utilities. CSS-in-JS adds a library abstraction on top of an abstraction.

Alternatives considered: Tailwind (popular, fast); CSS Modules (scoped, no globals); styled-components.

Consequence: We write plain CSS in .css files imported into components. Styles will be simple and functional, not beautiful. That is intentional.

ADR-004 — Docs are static HTML (no build step) Pre-curriculum

Decision: All documentation files are static HTML with inline or linked CSS. No MDX, no Docusaurus, no static site generator.

Session: Pre-curriculum (curriculum design)

Why: Docs must be openable by double-clicking an HTML file in any OS browser. Zero dependencies. Zero build step. A future AI session can read the source directly. Students can edit docs without a tool chain.

Alternatives considered: Docusaurus (powerful but requires Node and a build); VitePress (same); plain Markdown (not viewable without a renderer).

Consequence: Each doc is a self-contained HTML file. One shared styles.css is referenced with a relative path. Internal links use relative paths only.

ADR-005 — Mock data before real APIs Pre-curriculum

Decision: All data used before Layer 7 is fake. Country data lives in /src/mock/ and is served by /src/services/. No real HTTP requests until Session 38.

Session: Pre-curriculum (curriculum design)

Why: Real APIs introduce: network errors, CORS, rate limits, response shape uncertainty, and loading/error state complexity — all at once. These are distractions when the goal is to learn component design and state. Mock data lets us control every variable.

Alternatives considered: Use a real free API from Session 1 (common in tutorials); mock with a library like msw.

Consequence: The UI must be buildable and fully demonstrable without an internet connection until Session 38. The service layer interface is designed so switching from mock to real data in Session 38 changes only one file.

New decisions will be added here as the project evolves. Every ADR must have a session reference.
