Components Are Functions

A React component is not magic — it is a JavaScript function that returns JSX. Session 04 taught you arrow functions and return values. Session 06 taught you import/export. Today you combine both: write your first real component function, CountryCard, in its own file and use it inside App.

2. Pre-Coding Quiz

Answer before reading the concept. Need 4/5 to proceed.

Question 1 of 5

In JSX, what is the difference between <div> and <CountryCard />?

Lowercase tags map to HTML elements. Capitalized tags map to your component functions. React calls the function and renders whatever JSX it returns.
Question 2 of 5

Why must a React component be named CountryCard (PascalCase) instead of countryCard?

This is a React rule, not a JavaScript rule. <countryCard /> would be interpreted as an unknown HTML element, not your function.
Question 3 of 5

What must a component function return?

A component is a function. Functions return values. A component's return value is JSX — React turns that into DOM updates. Same return pattern you learned in Session 04.
Question 4 of 5

Props in React are most like which JavaScript idea?

Props are how a parent passes data to a child component. Like function arguments, the child receives them but does not modify them. You will use props fully in Session 11.
Question 5 of 5

Why extract UI into a separate CountryCard component instead of writing all the JSX directly in App?

Components are reusable functions. Write the card layout once in CountryCard, then use <CountryCard /> wherever you need a card. App stays focused on page structure.

3. The Concept — JavaScript First, Then React

JavaScript: functions that return values (Session 04)

In Session 04 you learned that a function computes a value and returns it. The caller receives that value and uses it. 

function getCountryName() {
  return "Canada";
}

const name = getCountryName();  // name is "Canada"
console.log(name);              // prints: Canada

Arrow functions work the same way — with or without curly braces: 

const getRegion = () => "Americas";           // implicit return
const formatLabel = (text) => { return text }; // explicit return

React: a component is a function that returns JSX

A React component follows the exact same pattern. The only difference is what it returns: instead of a string or number, it returns JSX — a description of UI. 

function CountryCard() {
  return (
    <div className="country-card">
      <h2>Canada</h2>
      <p>Region: Americas</p>
    </div>
  );
}

PascalCase — React's naming rule

In JSX, React decides what something is by its first letter:

  • lowercase (<div>, <p>, <span>) → built-in HTML element
  • Capitalized (<CountryCard />, <App />) → your component function

Using a component: <CountryCard /> not CountryCard()

Inside JSX, you write component tags — you do not call the function yourself. React calls CountryCard() internally when it sees <CountryCard />. 

// Inside App's return — CORRECT
return (
  <div>
    <h1>Countries</h1>
    <CountryCard />
  </div>
);

// WRONG inside JSX — do not call it yourself
return (
  <div>
    {CountryCard()}  // works technically but breaks React patterns — avoid
  </div>
);

CountryCard anatomy

Today's component has three parts: the function, the return, and the export.

Composition — App uses CountryCard

App is also a component function. It composes smaller components the same way you nest HTML tags. Import CountryCard, then place it in App's JSX.

File import/export flow (Session 06)

Each component lives in its own file. The default export pattern from Session 06 is the standard for React components.

Props preview — function arguments (Session 11)

Today the country name and region are hardcoded inside CountryCard. Next session you will pass them as props — just like passing arguments to a function: 

// Session 11 preview — do NOT write this today
function CountryCard(props) {
  return (
    <div>
      <h2>{props.name}</h2>
      <p>Region: {props.region}</p>
    </div>
  );
}

// Parent passes data in:
<CountryCard name="Canada" region="Americas" />

Props are read-only. A component receives them; it does not reassign them. Think: function arguments you must not mutate.

Layer 1 → Layer 2 connection: Session 04 (arrow functions + return) + Session 06 (import/export) + Session 09 (JSX) = Session 10 (component function in its own file).

Common mistakes

// MISTAKE: lowercase component name
function countryCard() { … }  // JSX <countryCard /> won't work as intended

// MISTAKE: forgetting return
function CountryCard() {
  (<div>Canada</div>);  // returns undefined — blank screen
}

// MISTAKE: calling the component like a regular function in JSX
return CountryCard();  // avoid — use <CountryCard />

// MISTAKE: forgetting export
function CountryCard() { … }
// App.jsx: import fails — nothing exported from module

10. What Breaks If This Code Is Removed?

  • No CountryCard.jsx: App's import fails — Vite shows a module not found error. Nothing renders.
  • No export default: Import succeeds but gets undefined — React error: "Element type is invalid."
  • No return in CountryCard: Function returns undefined — React renders nothing for that component (blank area).
  • Lowercase name + lowercase JSX tag: Browser treats it as an unknown HTML element — your function never runs.
  • Remove <CountryCard /> from App: Page renders only the heading — the card UI disappears. The component file exists but is unused.
