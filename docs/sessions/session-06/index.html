Layer 1 Session 06 JavaScript Foundations

Imports and Exports

React applications are split across many files. Each component lives in its own file. Data lives in its own file. Services live in their own files. ES modules — imports and exports — are how all these files talk to each other. Without this session, you cannot understand how a React app is assembled.

Estimated time: 35–40 minutes

1. Learning Objective

  • Explain the difference between named exports and default exports
  • Write a named export and import it correctly (with curly braces)
  • Write a default export and import it (without curly braces)
  • Import multiple named exports in one statement
  • Explain why a file can only have one default export
  • Move the countries array into its own module and import it

2. Pre-Coding Quiz

Answer before reading the concept. Need 4/5 to proceed.

Question 1 of 5

File A has: export const PI = 3.14;
How does File B import it?

Named exports are imported with curly braces. The name inside the braces must match the exported name exactly. No curly braces is for default exports.
Question 2 of 5

File A has: export default function greet() { return "hi"; }
How does File B import it?

Default exports are imported without curly braces. You can name them anything you like — import myGreet from './fileA' would also work.
Question 3 of 5

How many default exports can a single file have?

A file can have exactly one default export. It represents the primary thing the file provides. You can have as many named exports as you want alongside it.
Question 4 of 5

File A exports: export const x = 1; export const y = 2;
How do you import both in one line?

Multiple named exports are imported together in one set of curly braces, comma-separated.
Question 5 of 5

You import a named export as a different name: import { PI as pi } from './math';
What is the local variable name you can use?

The as keyword creates an alias. PI is no longer available — only pi is. This is useful when two modules export something with the same name.

3. The Concept — ES Modules

Why modules exist

Without modules, every JavaScript file shares the same global scope. Variables collide, dependencies are unclear, and files cannot be tested in isolation. ES modules solve this: each file has its own scope, and you explicitly declare what you share.

Named exports

Exporting (math.js)

export const PI = 3.14159;

export function add(a, b) {
  return a + b;
}

export const multiply = (a, b) => a * b;

Importing

import { PI, add, multiply } from './math';

console.log(PI);           // 3.14159
console.log(add(2, 3));    // 5
console.log(multiply(4,5)); // 20

Rules for named exports:

  • Must use curly braces when importing
  • The name must match exactly (or use as to alias)
  • A file can have as many named exports as needed

Default exports

Exporting (CountryCard.jsx)

function CountryCard({ name }) {
  return name;
}

export default CountryCard;

Importing

// Name can be anything
import CountryCard from './CountryCard';
import Card from './CountryCard'; // also valid

Rules for default exports:

  • No curly braces when importing
  • The importer can choose any local name
  • Only one default export per file
React convention: Each component file has one default export — the component itself. Helper functions or constants in the same file use named exports. When you see import React from 'react' or import CountryCard from './CountryCard', those are default imports.

Mixing named and default in one file

// countries.js
export const REGIONS = ["Americas", "Asia", "Europe", "Africa"];

export function findById(countries, id) {
  return countries.find(c => c.id === id);
}

const countries = [
  { id: 1, name: "Canada",  region: "Americas", population: 38000000 },
  { id: 2, name: "Japan",   region: "Asia",      population: 125000000 },
  { id: 3, name: "Brazil",  region: "Americas",  population: 215000000 }
];

export default countries;
// main.js — import both types from one file
import countries, { REGIONS, findById } from './countries';

console.log(countries.length);  // 3
console.log(REGIONS);           // ["Americas", "Asia", ...]
console.log(findById(countries, 2).name); // "Japan"

Import paths

// Relative path — file in same or nearby directory
import countries from './countries';        // same folder
import countries from '../data/countries';  // up one, then into data/

// Package import — from node_modules (no ./ prefix)
import React from 'react';
import { useState } from 'react';
In plain Node.js you may need .js extension. In Vite (our React setup, starting Session 08), extensions can be omitted for .js and .jsx files.

4. Lab

Lab objective: Move the countries array into src/data/countries.js as a default export. Add named exports for helper data and functions. Import and use them from a separate file.
1

Create the data directory and new file

Create src/data/countries.js (new directory, new file).

2

Add named exports and a default export

// src/data/countries.js

export const REGIONS = ["Americas", "Asia", "Europe", "Africa", "Oceania"];

export function findById(countries, id) {
  return countries.find(c => c.id === id);
}

export function filterByRegion(countries, region) {
  return countries.filter(c => c.region === region);
}

const countries = [
  { id: 1, name: "Canada",  region: "Americas", population: 38000000 },
  { id: 2, name: "Japan",   region: "Asia",      population: 125000000 },
  { id: 3, name: "Brazil",  region: "Americas",  population: 215000000 }
];

export default countries;
3

Create src/main.js to import and use it

// src/main.js
import countries, { REGIONS, findById, filterByRegion } from './data/countries';

console.log("Countries:", countries.length);
console.log("Regions:", REGIONS);
console.log("Find id 2:", findById(countries, 2).name);
console.log("Americas:", filterByRegion(countries, "Americas").map(c => c.name));
4

Run it

In your terminal, from the src/ folder (or using Node directly):

node src/main.js

If you get a module error, add "type": "module" to a package.json in the root, or rename to .mjs. In Session 08 this will be handled automatically by Vite.

5

Try importing the default under a different name

// This is valid — default exports can be named anything
import allCountries from './data/countries';
console.log(allCountries === countries); // true if same import — same module instance

This shows that the name of a default import is local — it's just a variable.

9. Reflection Questions

  1. A React component file typically has one default export (the component) and sometimes named exports (helper functions). Why do you think that convention exists?
  2. When you write import { useState } from 'react', is useState a named or default export from the React package?
  3. If two different modules you're importing both export a function named format, how would you handle the collision?
  4. Predict the folder structure of a React app after Session 10. How many files will there be and what will each export?

10. What Breaks If This Knowledge Is Missing?

  • Session 08 onward: Every React file uses imports. Without understanding named vs default, you will see import errors constantly and not know why.
  • The curly brace mistake: Trying to import a default export with curly braces (import { CountryCard }) gives undefined or an error. This is the most common import mistake.
  • Session 28 (Testing): Tests import the component under test. Understanding default imports is required.
