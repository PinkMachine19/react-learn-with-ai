3. The Concept — JavaScript First, Then React

JavaScript: The DOM as a tree of objects

When the browser loads a page, it parses HTML and builds the Document Object Model (DOM) — a live tree of objects in memory. Every tag becomes a node. Nodes have parents and children. You have been using this tree implicitly whenever JavaScript touched a page. 

// The browser built this tree from HTML:
// document
//   └── html
//         ├── head
//         │     └── title  →  text: "My Page"
//         └── body
//               └── h1     →  text: "Hello"
//               └── p      →  text: "World"

console.log(document.body.children.length);  // how many direct children body has
console.log(document.querySelector("h1").textContent);  // "Hello"

Key idea: the DOM is not the HTML file on disk. It is a mutable, in-memory object tree the browser maintains. Change a node and the page updates immediately.

JavaScript: Imperative DOM updates

Vanilla JavaScript updates the UI imperatively — you write explicit steps to find elements and mutate them. This works for small pages but becomes fragile as UI grows. 

// Imperative: HOW to update the page
const titleEl = document.querySelector("#title");
const countEl = document.querySelector("#count");
let count = 0;

document.querySelector("#btn").addEventListener("click", () => {
  count++;
  countEl.textContent = String(count);       // step 1: update count text
  if (count > 5) {
    titleEl.textContent = "High score!";     // step 2: maybe update title
    titleEl.classList.add("winner");         // step 3: maybe add a class
  }
});
// You must remember every element that depends on count.
// Miss one? UI is out of sync with state.

React: A library for building UI

React is a JavaScript library (not a full framework) for building user interfaces. It does not replace the DOM — it manages updates to it. React gives you:

  • A component model — UI split into reusable functions (Session 10)
  • A declarative render model — describe UI as a function of state
  • An efficient update mechanism — the virtual DOM diff (below)

React sits on top of everything you learned in Layer 1: functions, objects, arrays, map(), arrow functions, destructuring, imports/exports, and async data fetching.

React: Declarative model — UI = f(state)

In React you do not manually find and update DOM nodes. You describe what the UI should look like given the current state. When state changes, React re-runs your component function and reconciles the result with what's on screen. 

// Declarative mental model (conceptual — full JSX in Session 09)
function Counter({ count }) {
  return (
    <div>
      <h1 className={count > 5 ? "winner" : ""}>
        {count > 5 ? "High score!" : "Keep clicking"}
      </h1>
      <p>{count}</p>
    </div>
  );
}
// Change count → React re-renders → DOM updates automatically.
// You never call querySelector.

React: Virtual DOM (simplified)

Updating the real DOM is expensive. React keeps a lightweight virtual DOM — a JavaScript object tree describing what the UI should look like. On each render:

  1. React builds a new virtual tree from your component output
  2. It diffs the new tree against the previous one
  3. It applies only the minimal set of changes to the real DOM

You do not need to implement this — React handles it. But understanding why React exists helps you debug weird UI behaviour later.

Vite: dev server, JSX bundling, scaffolding

Browsers cannot run JSX. Vite is the build tool that transforms your React source files into plain JavaScript the browser understands, serves them on a local dev server, and hot-reloads changed files without a full page refresh. 

# Create a React project in the current directory
npm create vite@latest . -- --template react

# Install dependencies
npm install

# Start the dev server
npm run dev

Vite handles:

  • Dev server — serves files at http://localhost:5173 (default)
  • JSX transform — converts <App /> into React.createElement(...)
  • HMR — Hot Module Replacement; save a file, see changes instantly
  • Production build — npm run build (later sessions)

Project structure: index.html, main.jsx, App.jsx

react-learn-with-ai/
├── index.html          ← single HTML shell; has <div id="root">
├── package.json        ← scripts: dev, build, preview
├── vite.config.js      ← Vite configuration
└── src/
    ├── main.jsx        ← entry point: mounts React to #root
    ├── App.jsx         ← root component (returns null today)
    └── App.css         ← styles (unused until we render UI)

index.html — the only HTML file. Contains <div id="root"></div> where React attaches.

main.jsx — JavaScript entry. Calls ReactDOM.createRoot and renders <App />.

App.jsx — your root React component. Today it returns null — intentionally nothing.

JSX overview (preview — deep dive in Session 09)

JSX looks like HTML inside JavaScript. It is syntax sugar for React.createElement(...) calls. Vite transforms it before the browser sees it. 

// These are equivalent (Session 09 proves this):
const element = <h1 className="title">Hello</h1>;

const element = React.createElement("h1", { className: "title" }, "Hello");

JSX rules preview (full list in Session 09):

  • One root element per return (or use a Fragment)
  • className not class; htmlFor not for
  • JavaScript expressions go inside {"{ expression }"}
  • Self-closing tags must close: <img />
Layer 1 connection: main.jsx uses ES module imports exactly like Session 06. App is a function — exactly like Session 04 arrow functions. Country data from src/data/countries.js will appear in components starting Session 10.

10. What Breaks If This Knowledge Is Missing?

  • Debugging blank screens: Without understanding the boot chain (index.html → main.jsx → App), a blank page looks like a failure instead of intentional success — or a real error goes unnoticed.
  • Imperative habits in React: Calling document.querySelector inside a React component fights the declarative model and causes bugs that are hard to trace.
  • Confusing React with Vite: JSX errors are Vite/build errors. Runtime bugs are React errors. Without separating the two, you debug the wrong tool.
  • Virtual DOM misconceptions: Thinking React "replaces the DOM" leads to confusion when you inspect real DOM nodes in DevTools and they look like normal HTML.
