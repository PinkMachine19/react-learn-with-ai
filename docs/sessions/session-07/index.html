2. Pre-Coding Quiz

Answer before reading the concept. Need 4/5 to proceed.

Question 1 of 5

What are the three states a Promise can be in?

A Promise starts as pending. It either fulfills (succeeds) or rejects (fails). Once settled, it never changes state again.
Question 2 of 5

What does await do inside an async function?

await pauses only that async function — the rest of JavaScript continues running. When the Promise settles, execution resumes at that point. It does NOT block the thread.
Question 3 of 5

An async function always returns:

Marking a function async wraps its return value in a Promise. If you return 42 from an async function, callers receive Promise<42>, not 42 directly.
Question 4 of 5

How do you handle errors from an awaited Promise?

With async/await you use try/catch exactly like synchronous error handling. If the awaited Promise rejects, execution jumps to the catch block.
Question 5 of 5

This code runs. What does the console print, and in what order? 

console.log("A");
setTimeout(() => console.log("B"), 0);
console.log("C");

setTimeout (even with 0ms) pushes the callback to the task queue. JavaScript finishes the synchronous code (A, C) first, then pulls B from the queue. This is the event loop.

3. The Concept — Asynchronous JavaScript

Why asynchronous code exists

JavaScript runs on a single thread. If fetching data blocked the thread, the browser would freeze — no scrolling, no clicks, nothing. The solution: start the work, hand it off, continue running, and pick up the result when it's ready. That handoff mechanism is a Promise.

The event loop (simplified)

1. Synchronous code runs first

Your script top-to-bottom: variable declarations, function calls, console.logs.

2. Async work is delegated

fetch(), setTimeout(), file reads — these are handed to the browser/Node runtime.

3. Callbacks queue up

When the async work finishes, its callback or Promise resolution waits in the queue.

4. Event loop picks them up

Once the call stack is empty, the event loop picks the next item from the queue and runs it.

Promises — the primitive

const p = new Promise((resolve, reject) => {
  setTimeout(() => {
    resolve("data is ready");  // or: reject(new Error("something broke"))
  }, 1000);
});

p.then(data => console.log(data))   // "data is ready" — after 1 second
 .catch(err => console.error(err));  // only runs if rejected

Promise states:

  • pending — initial state, neither fulfilled nor rejected
  • fulfilled — resolve(value) was called; .then() fires
  • rejected — reject(reason) was called; .catch() fires

async/await — syntactic sugar over Promises

Promise chain style

function getCountries() {
  return fetch('/api/countries')
    .then(res => res.json())
    .then(data => data)
    .catch(err => console.error(err));
}

async/await style (same thing)

async function getCountries() {
  try {
    const res = await fetch('/api/countries');
    const data = await res.json();
    return data;
  } catch (err) {
    console.error(err);
  }
}

Rules:

  • await can only be used inside an async function
  • async functions always return a Promise (even if you return a plain value)
  • await unwraps the Promise value — gives you the resolved data directly
  • If the Promise rejects, await throws — caught by try/catch

Fake async — Promise.resolve and setTimeout

// Simulates a network call without a real API
function fakeDelay(ms) {
  return new Promise(resolve => setTimeout(resolve, ms));
}

async function fetchCountries() {
  await fakeDelay(300);  // pretend network latency
  return [
    { id: 1, name: "Canada",  region: "Americas", population: 38000000 },
    { id: 2, name: "Japan",   region: "Asia",      population: 125000000 },
    { id: 3, name: "Brazil",  region: "Americas",  population: 215000000 }
  ];
}

// Calling it:
fetchCountries().then(countries => console.log(countries));
// or inside another async function:
// const countries = await fetchCountries();
React connection: In Session 32 (useEffect + fetch), you will call a real API from inside a useEffect. The pattern is identical: async function load() { const data = await fetch(...); setCountries(data); }. The fake async service you build now trains the muscle memory for that pattern.

Common mistakes

// MISTAKE: await outside async function — syntax error
function bad() {
  const data = await fetchCountries();  // SyntaxError
}

// MISTAKE: forgetting await — gets a Promise object, not the data
async function alsoWrong() {
  const data = fetchCountries();  // data is Promise<Array>, not Array
  console.log(data.length);       // undefined
}

// CORRECT
async function correct() {
  const data = await fetchCountries();  // data is Array
  console.log(data.length);            // 3
}

4. Lab

Lab objective: Create src/services/countryService.js — a fake async data layer that wraps the country array in a simulated delay. Add named and default exports. Call it from a main.js using async/await with error handling.
1

Create src/services/ directory and countryService.js

2

Write the service module

// src/services/countryService.js
import countries from '../data/countries';

function delay(ms) {
  return new Promise(resolve => setTimeout(resolve, ms));
}

export async function fetchAllCountries() {
  await delay(300);
  return [...countries];  // return a copy, not the original array
}

export async function fetchCountryById(id) {
  await delay(150);
  const found = countries.find(c => c.id === id);
  if (!found) throw new Error(`Country with id ${id} not found`);
  return found;
}

export async function fetchCountriesByRegion(region) {
  await delay(200);
  return countries.filter(c => c.region === region);
}

const countryService = { fetchAllCountries, fetchCountryById, fetchCountriesByRegion };
export default countryService;
3

Update src/main.js to use the service

// src/main.js
import { fetchAllCountries, fetchCountryById, fetchCountriesByRegion } from './services/countryService';

async function main() {
  try {
    const all = await fetchAllCountries();
    console.log("All countries:", all.map(c => c.name));

    const japan = await fetchCountryById(2);
    console.log("Country 2:", japan.name);

    const americas = await fetchCountriesByRegion("Americas");
    console.log("Americas:", americas.map(c => c.name));

    // Test error path
    const missing = await fetchCountryById(999);
    console.log("Should not reach here:", missing);
  } catch (err) {
    console.error("Caught:", err.message);
  }
}

main();
4

Run it

node src/main.js

Expected output (roughly — timing varies):

All countries: [ 'Canada', 'Japan', 'Brazil' ]
Country 2: Japan
Americas: [ 'Canada', 'Brazil' ]
Caught: Country with id 999 not found
5

Observe the async proof

Add this before calling main():

console.log("Before main()");
main();
console.log("After main() — this runs BEFORE main() finishes");

You should see Before main() and After main() — this runs... printed before the countries. This proves main() is non-blocking — the event loop continues while the delays run.

Remove those two extra logs after observing.

11. What Breaks If This Knowledge Is Missing?

  • useEffect: Data fetching in React happens inside useEffect with async functions. Without async/await, you'll see Promise objects instead of data in your UI.
  • Loading states: The non-blocking proof (data arrives later) is exactly why React has loading state patterns. You felt it in this lab.
  • Error handling: API calls fail. Without try/catch, a failed fetch crashes silently or shows a Promise rejection in the console and leaves users with a broken UI.
