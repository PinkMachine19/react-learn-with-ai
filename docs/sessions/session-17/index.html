Layer 3 Session 17 State

useState Hook

Session 16 explained why an ordinary let count = 0 cannot update the screen. Today we write the fix: the useState hook. CountryCard gets its first piece of real memory — a click counter that survives across renders and triggers a UI update every time it changes.

Estimated time: 35–40 minutes

1. Learning Objectives

  • Import and call useState to declare a piece of component state
  • Explain why useState returns an array of exactly two items — a value and a setter
  • Know that the argument to useState(0) is only used on the very first render
  • Call the setter to update state and understand this schedules a re-render
  • Describe what "closure capture" means for the count variable inside one render

2. Pre-Coding Quiz

Answer before reading the concept. Need 4/5 to proceed.

Question 1 of 5

What does useState(0) return?

useState always returns a two-item array: [currentValue, setterFunction]. Array destructuring names them.
Question 2 of 5

Why does const [count, setCount] = useState(0) work no matter what you name the variables?

Session 05: array destructuring unpacks by position. Object destructuring would unpack by name — useState deliberately returns an array so you can name the pair anything.
Question 3 of 5

CountryCard re-renders three times after mount. How many times does useState(0)'s argument actually get used to set the value?

The initial value seeds state on mount only. After that, React remembers the current value across renders regardless of what argument you write in the source.
Question 4 of 5

What does calling setCount(count + 1) actually do?

The setter is the signal Session 16 said was missing from plain let variables — it tells React "re-run this component, something changed."
Question 5 of 5

If a click handler could somehow run twice before the component re-renders, what value of count would each call see?

count is declared with const inside the function body — each render's count is frozen for that render (closure capture, Session 04's function scope idea extended).

3. The Concept — JavaScript First, Then React

JavaScript: a function that returns a pair

Nothing magic is happening syntactically — useState is just a function call whose return value happens to be a two-item array, unpacked with the array destructuring from Session 05:

function makePair() {
  return [1, function increment() { /* ... */ }];
}

const [value, bump] = makePair();
// value → 1
// bump  → the function

useState works the same way — it just also plugs into React's internal bookkeeping so the "current value" it hands back is whatever was last set, not always the initial argument.

React: declaring state inside a component

import { useState } from 'react';

function CountryCard({ name, region }) {
  const [count, setCount] = useState(0);

  return (
    <div className="country-card">
      <h2>{name}</h2>
      <p>Region:{region}</p>
      <p>Clicked {count} times</p>
      <button onClick={() => setCount(count + 1)}>Click me</button>
    </div>
  );
}

Every CountryCard that gets rendered has its own independent count. Four cards on screen means four separate calls to useState(0), each with its own memory. Clicking one card's button never changes another card's count — this is the same per-instance isolation you'd expect from four separate function calls.

Not fully explained yet: the onClick={() => setCount(count + 1)} line uses an inline arrow function as an event handler. That is enough JSX to make the counter work today. Session 18 slows down and explains exactly what onClick is, what the event object contains, and why you often extract handlers into named functions.

Common mistakes

// MISTAKE: calling the setter during render, not inside a handler — infinite loop
function CountryCard() {
  const [count, setCount] = useState(0);
  setCount(count + 1); // runs every render → re-renders forever
  ...
}

// MISTAKE: mutating instead of using the setter — no re-render happens
let count = 0;
count = count + 1; // React never finds out

// MISTAKE: forgetting the initial value's type stays consistent
const [count, setCount] = useState(0);
setCount("5"); // works but now count is a string — comparisons downstream break

8. Post-Coding Quiz

Question 1 of 5

Four CountryCard components are on screen. How many times was useState(0) called in total, right after mount?

Each rendered instance of a component owns its own state. Four cards means four independent count values.
Question 2 of 5

After clicking a card's button once, why does the paragraph showing "Clicked N times" actually change on screen?

This is the re-render loop from Session 16, now wired up for real: state change → setter call → re-render → new JSX → diff → DOM update.
Question 3 of 5

You reload the page after clicking a card three times. What does the count show?

useState is in-memory only. No persistence layer exists yet — that is a much later topic (localStorage, backend, etc.), not something this session covers.
Question 4 of 5

What would happen if you wrote useState(0) as useState(Math.random())?

The initial-value argument is only consulted on mount, matching what Question 3 of the pre-quiz covered.

9. Reflection Questions

  1. In your own words, why does useState need to return a setter function instead of just letting you reassign a variable?
  2. What would you expect to see on screen if two different CountryCard instances somehow shared one count variable? Why doesn't that happen here?
  3. Why is the initial value argument to useState only meaningful on the first render?

10. What Breaks If This Code Is Removed?

  • Remove useState import: Build error — useState is not defined.
  • Remove the setter call, keep a plain variable: Clicking the button does nothing visible — this is the exact Session 16 failure mode, now reproducible on purpose.
  • Call setCount directly in the component body (not inside the handler): Infinite re-render loop — the browser tab will hang or crash.
  • Remove the button: Count is declared but nothing ever changes it — dead state.

11. What React Concept Was Learned Today

Today you learned: useState gives a component its own memory that changes over time and triggers a re-render when updated.
