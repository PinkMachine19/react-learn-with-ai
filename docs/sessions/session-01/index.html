Before we can understand React props, component data, or state — we must deeply understand JavaScript objects. Everything in React passes data as objects.

2. Pre-Coding Quiz

Question 1 of 5

You have this code: const country = { name: "Canada", region: "Americas" };
What does country.region return?

Dot notation reads a property by name. country.region is the same as country["region"]. Both return "Americas".
Question 2 of 5

You write const a = { x: 1 }; then const b = a; then b.x = 99;
What is the value of a.x after this?

Objects are reference types. const b = a does not copy the object — it copies the memory address. Both a and b now point to the exact same object, so changing b.x changes a.x too. const prevents reassigning the variable itself, but not mutating what it points to.
Question 3 of 5

A property key is stored in a variable: const key = "population";
Which syntax correctly reads that property from country?

country.key looks for a property literally named "key" — it does not evaluate the variable. Bracket notation country[key] evaluates the expression inside the brackets first, so it looks up the property named "population". Use bracket notation whenever the property name is dynamic (stored in a variable).
Question 4 of 5

You call a function and pass it an object: printCountry(country).
Inside printCountry, the parameter is also called country.
The function adds a new property: country.capital = "Ottawa".
After the function returns, does the original object have a capital property?

When you pass an object to a function, you pass the reference, not a copy. The parameter inside the function points to the same object in memory. Any mutations inside the function — adding, changing, or deleting properties — affect the original. This is why React makes props read-only: if child components could mutate them, data would change unpredictably from anywhere.
Question 5 of 5

Which of these is a valid way to create an object with a property that does not exist yet, then add it afterward?

Option A reassigns the variable itself (c = ...), which is illegal with const. Option C declares without initializing, causing an error when you try to access .flag on undefined. Option B is correct: const prevents reassigning the variable, but you can still mutate the object it points to by adding properties with dot notation.

3. The Concept — JavaScript Objects

What is an object?

An object is a container that groups related data together under one name. Instead of having three separate variables for a country's data, you have one object that holds all of them. 

// Without objects — three unrelated variables
const countryName = "Canada";
const countryRegion = "Americas";
const countryPopulation = 38000000;

// With an object — one container, logically grouped
const country = {
  name: "Canada",
  region: "Americas",
  population: 38000000
};

The thing on the left of the colon is called a key (or property name). The thing on the right is the value. Together they are a key-value pair. You can have as many as you need.

Reading properties — dot notation vs bracket notation

There are two ways to read a property from an object:

const country = { name: "Canada", region: "Americas" };

// Dot notation — use this when you know the property name upfront
console.log(country.name);    // "Canada"
console.log(country.region);  // "Americas"

// Bracket notation — use this when the property name is dynamic
const key = "name";
console.log(country[key]);    // "Canada" — evaluates the variable first

// Bracket notation is also required for property names with spaces or special chars
const weird = { "my country": "Canada" };
console.log(weird["my country"]); // "Canada"

Objects are reference types — this is critical

In JavaScript there are two categories of values: primitives and objects.

Primitives (string, number, boolean, null, undefined) are copied when assigned. Objects are not copied — only the memory address (reference) is copied. 

// Primitives — a copy is made
let x = 5;
let y = x;
y = 99;
console.log(x); // still 5 — x and y are independent

// Objects — reference is shared, NOT copied
const a = { score: 5 };
const b = a;        // b now points to the SAME object as a
b.score = 99;
console.log(a.score); // 99 — because a and b are the same object in memory
Why this matters for React: React components receive their data as an object called props. React makes props read-only on purpose — if child components could mutate the object, then parent components would have their data changed from unpredictable places. Understanding references is why this rule exists.

Adding and updating properties

const country = { name: "Canada" };

// Add a new property
country.capital = "Ottawa";
console.log(country); // { name: "Canada", capital: "Ottawa" }

// Update an existing property
country.name = "Canada (updated)";
console.log(country.name); // "Canada (updated)"

// Delete a property
delete country.capital;
console.log(country); // { name: "Canada (updated)" }

Nested objects

Property values can themselves be objects:

const country = {
  name: "Canada",
  location: {
    continent: "North America",
    coordinates: { lat: 56.1304, lng: -106.3468 }
  }
};

console.log(country.location.continent);          // "North America"
console.log(country.location.coordinates.lat);   // 56.1304

Objects in React — the connection

Every React component receives exactly one argument: an object called props. When you later write <CountryCard name="Canada" region="Americas" />, React collects those attributes into one object and passes it to your component function: 

// What React actually passes to your component:
props = {
  name: "Canada",
  region: "Americas"
}

// Your component reads it exactly like reading from any object:
function CountryCard(props) {
  return props.name; // "Canada"
}

We are not writing React yet. But when we do, in Session 10, you will immediately recognise this pattern because you already understand it.

10. What Breaks If This Knowledge Is Missing?

  • React props bugs: If you believe objects are copied when passed, you will be confused when a child component's mutation affects the parent's data. React cannot fully prevent this in plain objects — it relies on you not mutating.
  • State update bugs (Layer 3): In Session 17, you will learn that React state must be replaced, not mutated. If you mutate state directly (e.g., state.count = 5 instead of setState({ count: 5 })), React will not re-render because the reference did not change. This is the single most common React bug for beginners.
  • Bracket notation: In Session 14, you will use .map() to render lists. The callback receives an object, and you will need to access properties dynamically. Missing bracket notation means you cannot do this.
