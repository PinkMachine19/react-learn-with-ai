Event Handlers

Today we slow down: extract a named handler function, inspect the event object React hands it, and add a second handler (a reset button) so CountryCard has more than one thing a user can do.

2. Pre-Coding Quiz

Answer before reading the concept. Need 4/5 to proceed.

Question 1 of 5

What is the difference between onClick={handleClick} and onClick={handleClick()}?

This is the single most common event-handler bug for beginners. Parentheses invoke the function right now, not on click.
Question 2 of 5

When React calls your click handler, what does it pass as the argument?

Every DOM event handler in React receives an event object as its first argument, whether or not you choose to use it.
Question 3 of 5

Why does React use a "synthetic" event instead of handing you the raw browser event directly?

Different browsers can have slightly different native event objects; React's SyntheticEvent normalizes that so your handler code doesn't need per-browser checks.
Question 4 of 5

You want a "Reset" button that sets the count back to 0, separate from the "Click me" button that increments it. What's the right approach?

Each button gets its own handler function doing one clear job — the same "one job per function" idea from Session 04.
Question 5 of 5

Extracting () => setCount(count + 1) into a named function handleClick() { setCount(count + 1); } changes what, functionally?

Naming a handler doesn't change what it does — it's the same logic, easier to read and reuse.

3. The Concept — JavaScript First, Then React

JavaScript: passing a function vs calling a function

function sayHi() {
  console.log("hi");
}

setTimeout(sayHi, 1000);   // passes the function — runs once, after 1s
setTimeout(sayHi(), 1000); // calls sayHi() immediately, passes undefined to setTimeout

This is plain JavaScript, nothing React-specific: a function name with no parentheses is a reference to the function. Add parentheses and you're invoking it right now, using whatever it returns instead. onClick works exactly the same way.

React: extracting a named handler

import { useState } from 'react';

function CountryCard({ name, region }) {
  const [count, setCount] = useState(0);

  function handleClick(event) {
    console.log(event.type, event.target);
    setCount(count + 1);
  }

  function handleReset() {
    setCount(0);
  }

  return (
    <div className="country-card">
      <h2>{name}</h2>
      <p>Region:{region}</p>
      <p>Clicked {count} times</p>
      <button onClick={handleClick}>Click me</button>
      <button onClick={handleReset}>Reset</button>
    </div>
  );
}

handleClick receives the SyntheticEvent as its first parameter automatically — React always passes it, whether or not the function declares a parameter to catch it. handleReset doesn't need the event at all, so it simply doesn't declare a parameter for it — that's allowed; JavaScript doesn't force you to use every argument.

What's actually on the event object

PropertyWhat it holds
event.typeThe kind of event, e.g. "click"
event.targetThe actual DOM element that was interacted with — here, the <button>
event.preventDefault()A method to stop a browser's default behavior (not needed for a plain button, but essential later for form submissions)
Not in scope yet: typing in an input fires an onChange event using this exact same pattern — a handler function receiving an event object. We are not wiring up an input today; that's Session 20 (Controlled Inputs). Today is strictly about understanding the click path we already built in Session 17.

8. Post-Coding Quiz

Question 1 of 5

If you accidentally wrote onClick={handleReset()} instead of onClick={handleReset}, when would count reset to 0?

The parentheses invoke handleReset while React is building the JSX, not in response to a user click.
Question 2 of 5

handleReset is declared as function handleReset() with no parameters, yet React still calls it with a SyntheticEvent argument when the button is clicked. What happens to that argument?

Extra arguments a function doesn't declare parameters for are just ignored — ordinary JavaScript function-call behavior.
Question 3 of 5

Why does event.target inside handleClick refer to the <button> element specifically?

event.target always points to the specific element the interaction happened on.
Question 4 of 5

Why is React's event a "SyntheticEvent" rather than the browser's raw native event object?

Matches the pre-quiz explanation: normalized cross-browser behavior is the actual reason it exists.

10. What Breaks If This Code Is Removed?

  • Parentheses added back to onClick: Handler fires immediately during render instead of on click — often causes an infinite re-render loop when combined with a state setter.
  • Remove the event parameter from handleClick: The click still works (setCount still runs), but the console.log line breaks because event is undefined.
  • Remove handleReset and its button: No way to zero the counter without reloading the page.
  • Reuse handleClick for the Reset button: Clicking Reset would increment count instead of zeroing it — wrong behavior from sharing a handler that does the wrong job.

11. What React Concept Was Learned Today

Today you learned: event handlers are plain functions passed by reference to props like onClick, and React calls them with a normalized SyntheticEvent object.
