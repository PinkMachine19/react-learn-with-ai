Layer 2 Session 11 React Fundamentals

Props — Passing Data Down

Session 10 gave you a CountryCard component with hardcoded Canada and Americas. Today you turn those values into props — read-only inputs from the parent — so the same component can show any country without rewriting its JSX structure.

Estimated time: 35–40 minutes

1. Learning Objectives

  • Explain that props are read-only inputs — like function arguments
  • Pass data from a parent with JSX attributes: <CountryCard name="Canada" />
  • Receive props as a single object parameter: function CountryCard(props)
  • Read values with props.name and props.region
  • Destructure props in the parameter list: ({ name, region }) (Session 05)
  • Render two or more CountryCard instances with different prop values
  • State the one-way data flow rule: parent owns data; child displays it

Question 1 of 5

Props in React are most like which JavaScript idea?

The parent calls the child (via JSX). Props are the arguments. The child uses them; it does not invent or reassign them.
Question 2 of 5

JSX attributes become fields on one props object. Session 01 objects are the foundation.
Question 3 of 5

Why must a child component treat props as read-only?

Data flows parent → child. If the child needs to change something, that is state (later sessions) — not mutating props.

Session 05 destructuring applied to the props parameter — same idea as const { name } = country.
Question 5 of 5

3. The Concept — JavaScript First, Then React

JavaScript: parameters are inputs from the caller

In Session 04 you passed arguments into functions. The function did not invent those values — the caller supplied them. 

function greet(name) {
  return "Hello, " + name;
}

greet("Canada");  // "Hello, Canada"
greet("Japan");   // "Hello, Japan"

Same function body. Different inputs. Different outputs. That is the mental model for props.

React: JSX attributes become a props object

When App writes <CountryCard name="Canada" region="Americas" />, React calls your function with one argument — an object: 

// What React effectively does:
CountryCard({ name: "Canada", region: "Americas" });

Inside the component you read those fields and put them in JSX with curly braces (Session 09 — expressions in JSX): 

function CountryCard(props) {
  return (
    <div className="country-card">
      <h2>{props.name}</h2>
      <p>Region: {props.region}</p>
    </div>
  );
}

Destructuring props (Session 05)

Writing props.name everywhere works, but React code usually destructures in the parameter list: 

// props object style
function CountryCard(props) {
  return (
    <div>
      <h2>{props.name}</h2>
      <p>{props.region}</p>
    </div>
  );
}
// destructured style (preferred)
function CountryCard({ name, region }) {
  return (
    <div>
      <h2>{name}</h2>
      <p>{region}</p>
    </div>
  );
}

Both receive the same object. Destructuring pulls fields out so the body reads like plain variables. Same pattern as const { name } = country from Session 05.

Reuse: one component, many instances

Once props are in place, App can render multiple cards with different data: 

<CountryCard name="Canada" region="Americas" />
<CountryCard name="Japan" region="Asia" />
<CountryCard name="Brazil" region="Americas" />

Same CountryCard.jsx file. Three different prop objects. Three cards on screen. That is why components exist.

Layer 1 → Layer 2 connection: Session 01 (objects) + Session 04 (function parameters) + Session 05 (destructuring) + Session 09 (JSX expressions) + Session 10 (component function) = Session 11 (props).

Common mistakes

// MISTAKE: mutating props
function CountryCard(props) {
  props.name = "X";  // never — parent owns this data
}

// MISTAKE: forgetting curly braces for dynamic values
<h2>props.name</h2>   // shows the text "props.name"
<h2>{props.name}</h2> // shows Canada

// MISTAKE: treating props as separate parameters
function CountryCard(name, region) { … }  // wrong — one object only

// MISTAKE: hardcoding still, ignoring props
function CountryCard({ name }) {
  return <h2>Canada</h2>;  // always Canada — props unused
}

4. Lab

Lab objective: Update CountryCard to accept name and region as props. Pass those values from App.jsx. Render at least two cards with different countries.
1

Start from Session 10

You should already have src/components/CountryCard.jsx with hardcoded Canada / Americas, and App.jsx rendering <CountryCard />.

2

Update CountryCard.jsx to receive props

function CountryCard({ name, region }) {
  return (
    <div className="country-card">
      <h2>{name}</h2>
      <p>Region: {region}</p>
    </div>
  );
}

export default CountryCard;

Curly braces around name and region in JSX insert the prop values. Without braces, you would see the literal text name.

3

Pass props from App.jsx

import CountryCard from './components/CountryCard';

function App() {
  return (
    <div>
      <h1>Countries</h1>
      <CountryCard name="Canada" region="Americas" />
      <CountryCard name="Japan" region="Asia" />
    </div>
  );
}

export default App;

Run the dev server and verify

npm run dev

Expected in the browser:

  • Heading: Countries
  • A card for Canada / Region: Americas
  • A card for Japan / Region: Asia

Change one prop in App (for example name="Brazil") and confirm only that card updates — without editing CountryCard.jsx.

10. What Breaks If This Code Is Removed?

  • No props on CountryCard: You are back to Session 10 — every card shows the same hardcoded country.
  • No attributes in App: name and region are undefined — blank headings or "Region: undefined".
  • No curly braces in JSX: Screen shows the literal words name and region.
  • Mutating props: Parent and child disagree about the data; bugs become hard to track (React may warn in development).
  • Only one CountryCard: Lab incomplete — you never proved reuse with different prop values.
