Keys — Why React Needs Them

When lists change order or length, React must know which item is which. The key prop gives each list element a stable identity for reconciliation. This session completes Layer 2 — score ≥ 80% on both quizzes before starting Layer 3.

Estimated time: 35–40 minutes

3. The Concept — JavaScript First, Then React

JavaScript: identity in a list of objects

Session 02: each country has an id. If you reorder the array, the object with id: 2 is still Japan — its identity did not change, only its position did.

const a = { id: 1, name: "Canada" };
const b = { id: 2, name: "Japan" };
// id identifies the record — index is just current position

React: reconciliation (simplified)

On re-render, React builds a new element tree and compares it to the previous one. For lists, it asks: "Is this the same item as before, or a new one?"

  • With stable keys: React updates the correct card when data changes.
  • Without keys: React guesses by position — can reuse wrong DOM/state when items shift.
  • Index as key: Acceptable for static lists that never reorder. Risky when deleting or inserting at the start.

Adding keys to CountryList

{countries.map((country) => (
  <CountryCard
    key={country.id}
    name={country.name}
    region={country.region}
  />
))}

Remove all keys and read the browser console — React warns each child in a list should have a unique key prop.

Not in scope yet: State inside list items (Session 17+) makes bad keys painfully visible — wrong row keeps old state. Today focus on the warning and the identity idea.

10. What Breaks If This Code Is Removed?

  • No keys: Console warnings; subtle bugs when lists change (worse once state exists in items).
  • Duplicate keys: React cannot distinguish siblings — unpredictable updates.
  • Random keys: Every render looks like a new list — full remount, lost focus/state.
